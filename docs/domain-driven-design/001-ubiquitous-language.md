## How to use this glossary

- New domain terms must be added here **in the same change** that introduces them
  (CLAUDE.md). PRD-001 wins on any conflict — fix this doc, not the PRD.
- Code identifiers follow the todos-for-dues naming style: snake_case plural tables
  (`users`, `user_role_transitions`), snake_case columns, PascalCase TypeScript types,
  camelCase helpers, SCREAMING_SNAKE env vars. Identifiers for not-yet-built tables are
  prescriptive: designs use them or amend this row in the same change.
- `R-NN` / `AC-NN` / `Q-NN` references are PRD-001 IDs; `BC-NN` references are DDD-002.

## Identity & access

| ID | Term | Definition | Code identifier | Notes |
|----|------|------------|-----------------|-------|
| T-01 | User | A person known to the app. A row is auto-created in the Default Role (T-48) on first successful Authentik login — no approval gate (R-03). | `users` | Better Auth schema + `role_id` FK (was a `role` enum column, superseded by ADR-012); displayName/email from OIDC claims (AC-02). |
| T-02 | Role | **Amended by ADR-012:** an admin-managed, DB-backed named group with an editable app set (T-46). A User has **exactly one** (`users.role_id`). No longer a `Member`/`Admin` text enum — that is gone. Two seeded System Roles (T-47): **Admin** (T-04, superuser) and **Default** (T-48, new-user role); admins create more. Role assignment writes Audit Rows (`user_role_transitions`) in the same transaction (R-04). | `roles`, `users.role_id` | The old `enums.ts` `ROLES` array and `users.role` CHECK are removed (migration 0007). Session carries `role = { id, name, isAdmin }`. |
| T-03 | ~~Member~~ | **Superseded by ADR-012:** there is no `Member` role. The role every new login lands in is now the **Default** system role (T-48); "basic user" = a user in the Default role. | ~~`'Member'`~~ | Retained for history; use Default (T-48) / Role (T-02). |
| T-04 | Admin | **Amended by ADR-012:** the **superuser Role** (`roles.is_admin`) — implicitly sees every App (no `role_app_grants` rows; new Apps included automatically), grants admin-console access, and is immutable (no rename/edit/delete). First claimed via Bootstrap Admin (T-06); the last Admin-role member can't be moved off it (LastAdminError). | `roles.is_admin` (session `role.isAdmin`) | PRD-001 Actors & roles; R-02. `adminProcedure` gates on `role.isAdmin`. |
| T-05 | Family | **Amended by ADR-012:** no longer a flag/designation — it is an example Role (T-02), now **seeded by migration 0007** as a normal (editable/deletable) role granting every App except Tautulli. Phase-3 Family-Only-Library gating (R-26/R-27) will attach to a **role attribute**, not the removed `users.is_family`. | seeded row in `roles` (name `Family`; `is_admin`/`is_default`/`grants_all` all false) | `users.is_family` / `tags.is_family` are dropped (migration 0007); see ADR-012 C-09/C-12 and T-21. |
| T-06 | Bootstrap Admin | Idempotent promotion to Admin of a first login whose email (case-insensitive) is on the allowlist; repeat logins are no-ops (R-02, AC-03). | `BOOTSTRAP_ADMIN_EMAILS` (env) | Allowlist seeded with both owner emails (Q-01). |
| T-07 | Session | The authenticated browser session established after the Authentik OIDC round-trip; 7-day cookie (AC-01). | `sessions` (Better Auth) | No password form exists anywhere (R-01). |
| T-08 | Authentik | The OIDC identity provider at `authentik.haynesnetwork.com` — the only sign-in path. Plex is offered as the credential inside Authentik (R-01). | `AUTHENTIK_*` (env) | Syncing permissions into Authentik is a follow-on (R-30). |
| T-46 | Role App Grant | (ADR-012) A `(role, app)` pair — the editable app set a Role grants (T-13). The Admin superuser role holds **no** rows here (implicit all-apps). Composite PK dedupes; both FKs cascade. | `role_app_grants` | Replaces `user_app_grants` + `tag_app_grants` + `app_catalog.default_visible`. |
| T-47 | System Role | (ADR-012) One of the two seeded, fixed-id, non-deletable Roles: **Admin** (T-04, `is_admin`) and **Default** (T-48, `is_default`). Partial unique indexes enforce at most one of each; a CHECK forbids a role being both. Migration 0007 seeds a **third** role, `Family` (T-05), but it is a *normal* role, not a System Role — editable/deletable. | `roles.is_admin` / `roles.is_default` (`SEEDED_ROLE_IDS`) | Fixed ids so `users.role_id` can DB-default to Default. Domain writers reject renaming/deleting them (SystemRoleImmutableError). |
| T-48 | Default (Role) | (ADR-012) The System Role assigned to every new User (`users.role_id` DB default). Its seeded app set is **Seerr, Plex, K8Plex, and PlexOps** (T-10, migration 0007); the app set and description are editable but the role can't be renamed or deleted. | `roles.is_default` | The successor to the old "Member" default (T-03). |
| T-49 | All-Apps Role | (ADR-012, shipped after) A non-Admin Role with `roles.grants_all` = true: grants **every** catalog App — including ones added later — and holds **no** Role App Grants (T-46), i.e. Admin's (T-04) all-apps reach **without** admin-console access. Effective Permissions (T-16) = all apps when `is_admin OR grants_all`. Surfaced as the **"All apps"** checkbox on `/admin/roles` (greys out + disables the per-app list). | `roles.grants_all` | `createRole`/`updateRole` take `grantsAll` (true clears the explicit app set); `RoleInput`/`RolePatchInput` carry it; `roles.list` returns it. |

## Catalog & entitlements

| ID | Term | Definition | Code identifier | Notes |
|----|------|------------|-----------------|-------|
| T-09 | App (Catalog Entry) | A DB-backed catalog row for a self-hosted app: name, description, icon, URL, default-visible flag, display order (R-11). **Amended by ADR-013:** the URL is any well-formed, normalized `http(s)` URL — the host allow-list is gone. | `app_catalog` | Admin-editable without redeploy; seeds per R-12/R-13. The domain's `normalizeCatalogUrl` canonicalizes on write; DB backstop is the scheme-only CHECK `app_catalog_url_scheme` (`^https?://`, migration 0008). |
| T-10 | Default-Visible | **Amended by ADR-012:** no longer a per-App flag — "default visible" is now the **Default Role's app set** (T-46/T-48), seeded to Seerr, Plex, K8Plex, and PlexOps (R-12, migration 0007). Admins change what basic users see by editing the Default role's apps. | Default role's `role_app_grants` (was `app_catalog.default_visible`, dropped) | A fresh user (Default role) Dashboard shows exactly these (AC-04). |
| T-11 | Tile | The Dashboard rendering of one App the viewer is permitted to open; links out to its configured URL (R-10). | `Tile` (UI component) | |
| T-12 | Dashboard | The permissioned home page: the viewer's Tiles, derived from Effective Permissions. | `Dashboard` (route `/`) | Enforces (renders/hides); never decides — see DDD-002 §4. |
| T-13 | App Grant | **Amended by ADR-012:** permission to see/open one App is granted to a **Role**, not a User — a `(role, app)` pair (T-46). A User gets an App by being assigned a role that grants it. No per-user or tag-derived grants remain. | `role_app_grants` (was `user_app_grants`/`tag_app_grants`, dropped) | Role edits are audit-logged (`update_role`); role assignment via `user_role_transitions` (AC-05). |
| T-14 | ~~Tag~~ | **Superseded by ADR-012:** tags are removed. Their permission-bundle role is now filled by the **Role** itself (T-02) — a Role *is* the bundle. | ~~`tags`, `user_tags`~~ (dropped, migration 0007) | Read "tag" as "role"; bespoke access = a dedicated role. |
| T-15 | Permission Bundle | **Amended by ADR-012:** the bundle is now the **Role** (T-02): its App set (T-46) plus, since ADR-017, its allowed Plex libraries / family gating as role grants (R-20, R-25..R-27). | `role_app_grants` + `role_library_grants` | Assigning the role applies the bundle; one role per user. |
| T-16 | Effective Permissions | **Amended by ADR-012:** the Apps a User can open = **their Role's app set** (`role_app_grants`), or **ALL** catalog Apps if the Role is the Admin superuser **or an All-Apps Role** (T-49, `grants_all`). Single provenance (the role) — no union, no per-source attribution. Never stored; recomputed per query. | `EffectiveApp` (interface) + `effectiveAppsForUser()` | The `effective_app_grants` VIEW is dropped (migration 0007); `effectiveAppsForUser` returns all apps when `is_admin OR grants_all`. Changes take effect on the next dashboard query — no re-login (AC-05). |

## Plex servers & libraries

| ID | Term | Definition | Code identifier | Notes |
|----|------|------------|-----------------|-------|
| T-17 | Plex Server | One of the three Plex instances under management, keyed by owner slug: `haynestower` (legacy Unraid), `haynesops` and `hayneskube` (k8s) (R-25; OPS-002). Note the subdomain↔slug mismatch (plexops↔haynesops, k8plex↔hayneskube). | `plex_servers` (built, ADR-017; seeded by migration 0010) | Owner tokens from 1Password via `token_ref` (R-28); `machine_identifier` (T-54) keys the sharing API. |
| T-18 | Plex Library | A library on one specific Plex Server; **identity is `(server_id, section_key)`, never name** (Q-03 resolved: servers mirror Movies/TV under different names). | `plex_libraries` (built; UNIQUE(`server_id`,`section_key`)) | Soft-state `available` flag — a vanished library is withheld, not deleted (DESIGN-007 D-04). |
| T-19 | Library Grant | **Redefined by ADR-017:** a **Role → Library** grant — permission for members of a Role to self-add a given Plex Library (R-25..R-27). No longer per-user/tag (`library_grants`/`tag_libraries` retired before build). Mirrors T-46 Role App Grant. | `role_library_grants` (built) | Share application happens in Plex Sharing (BC-04); grant edits audited (`update_role_libraries`). |
| T-20 | Allowed Library Set | A User's effective set of self-addable libraries — the union of nothing (one Role per user): the User's Role's `role_library_grants`; Admin ⇒ all available libraries. `grants_all` does NOT imply all libraries (ADR-017 C-03). | `EffectiveLibrary` + `effectiveAllowedLibrariesForUser()` | Re-derived inside each share mutation (TOCTOU gate). |
| T-21 | Family-Only Library | **Redefined by ADR-017:** an ordinary Plex Library granted only to the `Family` Role — there is **no** `plex_libraries.is_family_only` column. Exactly `HNet Home Videos` and `HNet Photos` (HAYNESTOWER) today (R-26; C-09/C-12). | rows in `plex_libraries` referenced only by `role_library_grants` for the Family role | Family self-service is just its role's grant set. |
| T-52 | Plex Share | An applied per-user library share on a specific Plex Server — the user's `SharedServer` on the server owner's plex.tv account, holding the set of shared library sections. Add/remove mutates it via read-merge-write (never blind overwrite). | `plex_share_audit` (the append-only ledger of applied shares); plex.tv `shared_servers` | Precondition: the user is already a Plex friend of the owner (email match; ADR-017 C-06). |
| T-53 | Library Registry Refresh | The admin-triggered sync that reads each server's `GET /library/sections` (+ `/identity`) and upserts `plex_servers`/`plex_libraries` keyed on `(server_id, section_key)`, marking unseen libraries `available=false` (never deleting). **Degrades per server** (DESIGN-007 D-12): each server refreshes in its own transaction and a Plex failure on one is folded into a per-server summary (`{ ok, servers: [{ slug, name, ok, libraryCount?, error? }] }`) — it never aborts the others. | `refreshPlexRegistry` / `plex.refreshRegistry` | Admin-button only for now — a CronJob is future work (Q-06). NOT all-or-nothing: one unreachable server reports `error: 'unreachable'` while the rest still commit. |
| T-54 | Machine Identifier | A Plex Server's stable GUID; the key the plex.tv sharing API addresses a server by (`/api/servers/{machineIdentifier}/…`). Distinct from the owner slug and the ingress subdomain. | `plex_servers.machine_identifier` | Verified live 2026-07-06; pinned in `PLEX_MACHINE_IDENTIFIERS`, re-read by the registry refresh. |

| ID | Term | Definition | Code identifier | Notes |
|----|------|------------|-----------------|-------|
| T-22 | *arr | Collective for Sonarr (TV), Radarr (movies), Lidarr (music) — the Source of Truth for media lists (R-40). | `arr_kind` (`'sonarr' \| 'radarr' \| 'lidarr'`) | |
| T-23 | Monitored | *arr state meaning the *arr actively manages/searches for an item. The Ledger mirrors all monitored media (R-40). | `ledger_items.monitored` | *arr concept we reference. |
| T-24 | Media Item | One Ledger row mirroring a monitored *arr item: on-disk state, quality profile, root folder, *arr tags (R-40). | T-25 | Ledger | The app's durable, queryable mirror of the media estate: Media Items + Ledger Events + Wanted Items. | T-30 | Fix Reason | Required taxonomy value on every Fix Request: `wont_play_corrupt`, `wrong_language`, `wrong_version_quality`, `missing_subtitles`, `wrong_content`, or `other` + free text (R-45). | `fix_requests.reason`, `fix_requests.reason_text` | Stored for analysis. | T-31 | Release | *arr concept: a specific downloadable version/file of a media item offered by an indexer. Referenced, not owned. | — | |
| T-32 | Grab | *arr concept: the *arr taking a Release for download; appears in *arr history. Fix chooses its path by grab history (R-44). | — | |
| T-33 | Blocklist | *arr concept: marking a Release failed so the *arr won't re-grab it. Fix = mark-failed (blocklist) + trigger a new search (R-44, AC-07). | — | Via the *arr's history-failed endpoint. |
| T-34 | Fix Fallback | The Fix path when an item has no Grab history: delete the file(s) + trigger search (R-44, AC-08). | `fix_requests.path_taken = 'delete_search'`, `fix_requests.actions_taken` | Outcome recorded either way. |
| T-35 | Restore | The admin-only failsafe: diff the Ledger against a live *arr, preview, then re-add missing items Monitored with recorded quality profile, root folder, and tags (R-50..R-52). | `restore_runs` | The only bulk write-back; explicitly admin-initiated and audit-logged. |
| T-36 | Restore Preview | The pre-execution diff listing exactly the Ledger items absent from the target *arr (R-52, AC-09). | `RestorePreview` | Execution reports successes/failures per item. |

| ID | Term | Definition | Code identifier | Notes |

Sonarr episodes group into collapsible season sections in the UI. | `fix_requests.target_scope` (`item\|season\|episode\|album`) + `target_season`; `ledger_events.payload.scope` for Force Search | Validated by the shared `resolveFixTarget`/`resolveSearchTarget` (packages/domain `action-scope.ts`); commands: `SeasonSearch`/`ArtistSearch` added (DESIGN-005 D-15/D-17, migration 0006). |
| T-50 | Bazarr | The subtitle manager for the **Radarr/Sonarr estate** (`bazarr.media.svc.cluster.local:6767`); the Fix target for reason `missing_subtitles` (ADR-016). Covers movies + episodes, **not** music. Base path `/api`, auth header `X-API-KEY`. | `@hnet/arr` `BazarrClient`/`BazarrWriteClient`; `BAZARR_URL`/`BAZARR_API_KEY` | Async `search-missing` action (HTTP 204, queued); not an *arr — `assertBazarrEnv`, not `ARR_SERVICES`. |
| T-51 | Subtitle Fix | The Fix Path (`fix_requests.path_taken = 'bazarr_subtitle'`) taken when reason = `missing_subtitles`: no Blocklist (T-33), no Fix Fallback delete (T-34), no *arr re-grab — a Bazarr subtitle search only. Fire-and-forget: rests at `search_triggered` (never auto-completes; `completeFixRequests` excludes it). | `fix_requests.path_taken = 'bazarr_subtitle'` (migration 0009) | `runSubtitleFix` (packages/domain `fix-flow.ts`); movie → `PATCH /api/movies`, sonarr episode/season → series-level `PATCH /api/series` (ADR-016 / DESIGN-005 D-19). |
